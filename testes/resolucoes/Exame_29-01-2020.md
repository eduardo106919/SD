### Questão 02

Num sistema orientado a objetos, a invocação remota visa tornar a rede "transparente", fazendo com que uma chamada a um objeto noutra máquina pareça uma chamada local.

Componentes e Objetivos:
- Stub no Cliente: Atua como um representante local do objeto remoto. O seu objetivo é empacotar os argumentos e enviar o pedido pela rede.
- Dispatcher: Recebe o pedido no servidor e identifica qual o objeto e método específicos que devem ser executados, encaminhando a chamada.
- Skeleton no Servidor: Faz o processo inverso do Stub. Ele descompacta os argumentos, chama o método real no objeto servidor e empacota o resultado para enviar de volta.
- Módulo de Comunicação: Responsável pelo transporte físico dos bytes entre as máquinas, gerindo protocolos e conexões.
- Remote Reference Modules: Mantêm uma tabela para traduzir referências de objetos locais em referências remotas globais, garantindo que o cliente comunica com a instância correta no servidor.

### Questão 03

Uma Distributed Hash Table é um sistema distribuído descentralizado que fornece um serviço de procura semelhante a uma tabela de hash: pares (chave, valor). Os dados são espalhados por diversos nós, e qualquer nó pode encontrar o valor associado a uma chave de forma eficiente, geralmente com uma complexidade de saltos de `O(log N)`.

A necessidade de uma DHT surge para cumprir objetivos que arquiteturas centralizadas não conseguem manter em larga escala:
- Escalabilidade: Ao contrário de um servidor central, uma DHT não tem um gargalo. À medida que mais nós entram, a capacidade de armazenamento e processamento do sistema aumenta proporcionalmente.
- Tolerância a Falhas e Disponibilidade: Não existe um ponto único de falha. Se um nó sair da rede, o sistema redistribui as chaves e continua a operar.
- Descentralização/Partição: Permite que o sistema opere de forma autónoma sem uma autoridade central, facilitando o design de sistemas Peer-to-Peer globais onde os nós são dinâmicos.
