## Grupo 01

### Questão 01

Ao utilizar exclusão mútua (locks/mutexes), o objetivo é minimizar a contenção e o tempo que as threads passam bloqueadas. Dois cuidados essenciais são:
- Minimizar a Granularidade do Lock: Deve-se manter dentro da secção crítica apenas o código estritamente necessário que acede a recursos partilhados. Cálculos pesados ou operações de I/O que não dependem do recurso partilhado devem ser feitos fora do lock. Quanto menor for o tempo de posse do lock, maior é a probabilidade de outras threads avançarem sem esperar, aumentando o paralelismo real.
- Utilizar Primitivas de Leitura/Escrita (Reader-Writer Locks): Em muitos sistemas, as operações de leitura são muito mais frequentes que as de escrita. Um mutex padrão é exclusivo. Se utilizarmos locks de leitura/escrita, múltiplas threads podem ler o recurso simultaneamente de forma segura. A exclusão mútua só é aplicada quando uma thread precisa de escrever, o que reduz drasticamente o bloqueio desnecessário em cenários de "muitas leituras".

### Questão 02

A migração de código (como o uso de JavaScript num browser ou applets) justifica-se por vários motivos num sistema distribuído:
- Redução da Latência e Melhoria da Interatividade: Ao executar código localmente no cliente, evita-se o tempo de ida e volta ao servidor para tarefas simples, tornando a aplicação mais responsiva.
- Escalabilidade do Servidor: Ao "descarregar" o processamento para os milhares de clientes, o servidor poupa ciclos de CPU e memória, permitindo que suporte um número muito maior de utilizadores simultâneos com menos infraestrutura física.
- Redução do Tráfego de Rede: Em vez de enviar grandes volumes de dados brutos para o servidor processar e devolver, o cliente pode processar os dados localmente e enviar apenas o resultado final.

### Questão 03

Neste cenário de uma loja online com múltiplos sistemas heterogéneos (fornecedores diferentes) e localizações distintas, o problema mais crítico é a Atomicidade e Consistência Global (Transações Distribuídas).

**O Problema**: Como garantir que, se a cobrança for feita, a encomenda seja obrigatoriamente registada na expedição e o stock atualizado no armazém? Como estes sistemas são de diferentes fornecedores e estão em locais distintos, não existe uma base de dados única. Se um sistema falhar a meio do processo, o sistema pode ficar num estado inconsistente.

**Solução Típica**: Protocolo de Compromisso em Duas Fases (Two-Phase Commit): Um coordenador pergunta a todos os sistemas se estão prontos para efetivar a operação (Fase 1). Se todos disserem "sim", o coordenador ordena a finalização (Fase 2). Se um falhar, todos fazem rollback.
