### Questão 01

O algoritmo de Filtros é uma generalização do algoritmo de Peterson para n threads, onde cada "camada" funciona como um funil que reduz o número de threads que podem avançar.
- Por que n−1 camadas? Para garantir a exclusão mútua entre n threads, precisamos de n−1 camadas. Em cada camada, pelo menos uma thread fica retida. Assim, após passar por n−1 camadas, apenas uma única thread conseguirá entrar na Secção Crítica.
- Se usasse menos camadas: O sistema não garantiria exclusão mútua. Por exemplo, com n threads e apenas n−2 camadas, duas threads poderiam potencialmente chegar à Secção Crítica simultaneamente.
- Se usasse mais camadas: O algoritmo continuaria a funcionar corretamente, mas seria ineficiente. Adicionar camadas extras aumenta o overhead de processamento e a latência sem qualquer benefício adicional de segurança.


### Questão 02

**O Problema**: Existe uma violação da causalidade entre as mensagens (b) e (c), pois (c) tem um valor menor que (b), embora tenha ocorrido após (b).


**Explicação**:
- No processo p3​, o envio da mensagem (c) ocorre após o envio da mensagem (b). Pelas regras dos relógios de Lamport, o relógio deve ser incrementado a cada evento. Se (b) foi enviado com valor 5, a mensagem seguinte (c) deveria ter, obrigatoriamente, um valor superior (L(c)>5). No entanto, (c) tem o valor 4.
- Problema no envio de (d): p1​ envia a mensagem (d) após receber (b). Se p1​ recebeu (b) com valor 5, o seu relógio local deveria ter sido atualizado para 6 (`max(local,5)+1`). Assim, a mensagem (d) nunca poderia ter o valor 5; teria de ser pelo menos 6.


### Questão 03

O protocolo Two-Phase Commit (2PC) lida com falhas de participantes de forma rigorosa para garantir a atomicidade.
- Deteção: O Coordenador envia uma mensagem `VOTE_REQUEST` e inicia um timeout.
- Ação do Coordenador: Se um participante falhar ou não responder dentro do tempo limite, o Coordenador assume um voto negativo.
- Resultado: Como o 2PC exige unanimidade, se um único participante falhar na primeira fase, o Coordenador decide pelo Aborto Global e envia uma mensagem de `GLOBAL_ABORT` a todos os participantes que estão operacionais, garantindo que nenhum recurso seja alterado indevidamente.


### Questão 04

Neste cenário, temos um problema de Ponto Único de Falha e a necessidade de Eleição de Líder.
- O Problema Mais Importante: É a Disponibilidade e a Tolerância a Faltas. Se o dispositivo que serve de intermediário (o "líder") falhar ou perder a ligação WiFi, todos os outros dispositivos ficam isolados da cloud, tornando o sistema inoperante.
- Solução Típica: Implementação de um Algoritmo de Eleição (como o Bully Algorithm).
- Justificação: 1. Os dispositivos devem ser capazes de detetar a falha do intermediário atual (através de heartbeats). 2. Ao detetar a falha, o sistema inicia um processo de eleição para escolher automaticamente um novo intermediário entre os dispositivos ativos. 3. Isso garante que o sistema seja auto-regenerável, mantendo a conectividade com a cloud enquanto houver pelo menos um dispositivo com acesso à rede.
